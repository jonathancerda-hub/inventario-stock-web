1. Resumen Ejecutivo

1.1 Visión del Producto

Sistema web centralizado para visualización y análisis de inventario en tiempo real, integrado con Odoo ERP, que permite a usuarios autorizados monitorear stock, fechas de vencimiento y tomar decisiones basadas en datos actualizados.

1.2 Objetivos del Negocio

  • Reducir pérdidas por vencimiento: Identificación temprana de productos próximos a vencer (0-3 meses)
  • Centralizar información: Consolidar datos de múltiples ubicaciones en una sola interfaz
  • Mejorar eficiencia operativa: Reducir tiempo de consulta de inventario de 15 min a <30 segundos
  • Facilitar análisis: Dashboard con 6 gráficos interactivos y exportación a Excel
  • Control de acceso granular: Sistema RBAC con 4 roles y permisos específicos por funcionalidad

3. Requisitos Funcionales

3.1 Autenticación y Autorización

RF-001 - Autenticación OAuth 2.0 con Google
  • Descripción: El sistema debe autenticar usuarios mediante Google OAuth 2.0
  • Prioridad: ALTA
  • Criterios de aceptación:
    • Redirección automática a Google Login
    • Intercambio seguro de tokens OAuth
    • Validación de usuario en base de datos post-autenticación
    • Verificación de estado activo del usuario
    • Registro automático del último login
    • Creación de sesión persistente con rol del usuario (15 min de inactividad)
  • Implementación: Authlib 1.6.8, endpoints /google-oauth y /authorize
RF-002 - Control de Acceso por Base de Datos
  • Descripción: Solo usuarios registrados y activos en la base de datos pueden acceder al sistema
  • Prioridad: ALTA
  • Criterios de aceptación:
    • Usuarios registrados en tabla inventario_user_permissions (PostgreSQL/Supabase)
    • Validación de estado activo (is_active=true)
    • Registro automático del último login (last_login)
    • Mensaje de error claro para usuarios no registrados o inactivos
    • Log completo de intentos de acceso en tabla inventario_audit_log
  • Implementación: permissions_manager.get_user_details() y permissions_manager.update_last_login()
RF-003 - Autorización por Niveles (3 niveles)
  • Descripción: Control granular de permisos según rol de usuario
  • Prioridad: ALTA
  • Niveles:
    • Nivel 1 (General): /inventory, /exportacion
    • Nivel 2 (Dashboard): Nivel 1 + /dashboard
    • Nivel 3 (Admin): Nivel 2 + /analytics
  • Implementación: roles.json + decoradores @require_auth + @dashboard_required

    • Autenticación con credenciales de Odoo

4. Requisitos No Funcionales

4.1 Seguridad (OWASP Top 10 2021)

RNF-001 - A01:2021 - Broken Access Control
  • Mitigación:
    • OAuth 2.0 obligatorio para toda autenticación
    • Sistema RBAC con base de datos (inventario_user_permissions)
    • Decoradores @require_auth y @require_admin en todas las rutas protegidas
    • Validación granular de permisos por funcionalidad (permissions_manager)
    • Auditoría completa de accesos (inventario_audit_log)
    • Rate limiting: 60 requests/minuto por IP
RNF-002 - A02:2021 - Cryptographic Failures
  • Mitigación:
    • HTTPS obligatorio en producción (Flask-Talisman)
    • SESSION_COOKIE_SECURE=True
    • SESSION_COOKIE_HTTPONLY=True
    • SECRET_KEY de 32 bytes aleatorios (no hardcoded)
    • Passwords de Odoo en variables de entorno, nunca en código
RNF-003 - A03:2021 - Injection (SQL & XML)
  • Mitigación:
    • Queries parametrizadas en PostgreSQL (psycopg2)
    • Pydantic v2 para validación de inputs en 5 endpoints
    • defusedxml para protección contra XXE en XML-RPC
    • Validación de ANALYTICS_TABLE_PREFIX con regex (solo [a-zA-Z0-9_])
RNF-004 - A04:2021 - Insecure Design
  • Mitigación:
    • Arquitectura en capas con separación de responsabilidades
    • Validación centralizada (schemas.py)
    • Configuración centralizada (config.py)
    • Logging estructurado sin información sensible
RNF-005 - A05:2021 - Security Misconfiguration
  • Mitigación:
    • DEBUG=False en producción
    • CSP (Content Security Policy) configurado con Flask-Talisman
    • HSTS (HTTP Strict Transport Security) habilitado
    • X-Frame-Options: DENY
    • X-Content-Type-Options: nosniff
    • Permissions-Policy para APIs sensibles deshabilitadas
RNF-006 - A07:2021 - Identification and Authentication Failures
  • Mitigación:
    • Session fingerprinting (SHA-256 de User-Agent + IP)
    • Timeout de sesión: 15 minutos de inactividad
    • Keep-alive endpoint para sesiones activas (/api/keep-alive)
    • Modal de advertencia a los 14 minutos
    • Logout automático por expiración
RNF-007 - A09:2021 - Security Logging and Monitoring Failures
  • Mitigación:
    • Logging estructurado con módulo logging
    • 4 niveles de log: DEBUG, INFO, WARNING, ERROR
    • Logs de autenticación (login exitoso/fallido)
    • Logs de acceso denegado
    • Analytics de visitas en base de datos persistente
    • Retención de logs: 30 días

    • Caché LRU de datos del dashboard

  • Usuarios concurrentes: Hasta 50 simultáneos

6. Casos de Uso Principales

Caso de Uso 1: Login con Google OAuth

Actor: Usuario final
Precondición: Usuario está registrado y activo en la base de datos
Flujo principal:
  1. Usuario accede a /login
  2. Usuario hace clic en "Iniciar sesión con Google"
  3. Sistema redirige a Google OAuth
  4. Usuario autoriza en Google
  5. Google redirige a /authorize con token
  6. Sistema consulta usuario en tabla inventario_user_permissions
  7. Sistema valida que is_active=true
  8. Sistema registra último login (last_login)
  9. Sistema crea sesión persistente con rol del usuario
  10. Sistema redirige según permisos: /dashboard si tiene view_dashboard, sino /inventory
Flujo alternativo: 6a. Usuario NO registrado o inactivo → Sistema muestra mensaje "Acceso Denegado" y registra intento en audit_log
Postcondición: Usuario autenticado con sesión activa y rol asignado

Caso de Uso 2: Consultar Inventario con Filtros

Actor: Usuario estándar
Precondición: Usuario autenticado
Flujo principal:
  1. Usuario accede a /inventory
  2. Sistema muestra tabla con todo el inventario
  3. Usuario selecciona filtros (línea, categoría, ubicación)
  4. Usuario hace clic en "Buscar"
  5. Sistema valida filtros con Pydantic
  6. Sistema consulta datos filtrados desde Odoo
  7. Sistema aplica código de colores por vencimiento
  8. Sistema muestra tabla filtrada
Flujo alternativo: 5a. Filtros inválidos → Sistema muestra error de validación
6a. Odoo no responde → Sistema muestra mensaje "Error de conexión, intente más tarde"
Postcondición: Tabla de inventario filtrada visible

Caso de Uso 3: Ver Dashboard con KPIs

Actor: Usuario dashboard
Precondición: Usuario autenticado y con permiso de dashboard
Flujo principal:
  1. Usuario accede a /dashboard
  2. Sistema valida permiso de dashboard
  3. Sistema consulta datos de Odoo (con caché LRU)
  4. Sistema calcula KPIs (total productos, cantidad, vencimiento)
  5. Sistema genera 6 gráficos con Chart.js/ECharts
  6. Sistema muestra dashboard interactivo
Flujo alternativo: 2a. Usuario sin permiso → Redirect a /inventory con mensaje
3a. Datos en caché → Sistema muestra versión cacheada (5 min)
Postcondición: Dashboard con KPIs y gráficos visible

Caso de Uso 4: Exportar a Excel

Actor: Usuario estándar
Precondición: Usuario autenticado y ha aplicado filtros
Flujo principal:
  1. Usuario hace clic en "Exportar a Excel"
  2. Sistema valida filtros actuales
  3. Sistema consulta datos filtrados de Odoo (max 10k filas)
  4. Sistema genera archivo Excel con pandas/openpyxl
  5. Sistema formatea números con separador de miles
  6. Sistema envía archivo como descarga (inventario_YYYYMMDD_HHMMSS.xlsx)
  7. Navegador descarga archivo
Flujo alternativo: 3a. Más de 10k filas → Sistema muestra error "Demasiados resultados, refine filtros"
4a. Error generando Excel → Sistema muestra mensaje "Error al generar reporte"
Postcondición: Archivo Excel descargado localmente

Caso de Uso 5: Monitorear Analytics (Admin)

Actor: Administrador
Precondición: Usuario autenticado y con rol admin
Flujo principal:
  1. Admin accede a /analytics
  2. Sistema valida rol de admin
  3. Sistema consulta datos de Supabase PostgreSQL
  4. Sistema calcula métricas (visitas, usuarios únicos, etc.)
  5. Sistema genera gráficos de serie temporal
  6. Sistema muestra dashboard de analytics
Flujo alternativo: 2a. Usuario sin rol admin → HTTP 403 Forbidden
3a. Supabase no disponible → Sistema usa SQLite local (modo degradado)
Postcondición: Dashboard de analytics visible

Caso de Uso 6: Gestionar Usuarios y Roles (Admin Module v2.1)

Actor: Administrador (admin_full)
Precondición: Usuario autenticado con rol admin_full y permiso manage_users
Flujo principal:
  1. Admin accede a /admin/users
  2. Sistema valida permiso manage_users
  3. Sistema consulta inventario_user_permissions de Supabase
  4. Sistema muestra lista de usuarios con stats cards (Total, Admins, Cambios recientes)
  5. Admin puede filtrar por rol o buscar por email
  6. Admin selecciona una acción:
    • Crear usuario: Formulario con email, nombre, rol → Sistema valida @agrovetmarket.com → Sistema inserta en BD → AuditLogger registra user_created
    • Editar rol: Selecciona nuevo rol con vista previa de permisos → Sistema actualiza BD → AuditLogger registra role_changed
    • Eliminar: SweetAlert2 confirma → Sistema soft delete (is_active=FALSE) → AuditLogger registra user_deleted
    • Reactivar: Restaurar usuario eliminado → Sistema set is_active=TRUE → AuditLogger registra user_reactivated
  7. Sistema muestra mensaje de éxito con feedback visual
  8. Sistema actualiza tabla con DataTables (búsqueda en tiempo real, paginación)
Flujo alternativo: 2a. Usuario sin permiso manage_users → HTTP 403 Forbidden con mensaje "No autorizado"
6a. Email duplicado → Sistema muestra error "Usuario ya existe"
6b. Email inválido → Sistema muestra error "Debe ser @agrovetmarket.com"
6c. Admin intenta eliminarse a sí mismo → Sistema bloquea acción con mensaje "No puedes eliminarte a ti mismo"
6d. Supabase no disponible → Sistema muestra error "Error de base de datos, intente más tarde"
Postcondición: Usuario creado/modificado/eliminado en BD, auditoría registrada con IP + user agent, UI actualizada con feedback visual

Caso de Uso 7: Consultar Registro de Auditoría (Admin Module v2.1)

Actor: Administrador (admin_full)
Precondición: Usuario autenticado con rol admin_full y permiso manage_users
Flujo principal:
  1. Admin accede a /admin/audit-log
  2. Sistema valida permiso manage_users
  3. Sistema consulta inventario_audit_log de Supabase
  4. Sistema muestra tabla de auditoría con DataTables (action, user_email, performed_by, timestamp, details JSONB)
  5. Admin puede filtrar por:
    • Tipo de acción (user_created, role_changed, user_deleted, user_reactivated)
    • Usuario afectado (email)
    • Rango de fechas (últimos 7/30/90 días)
  6. Admin puede expandir details JSONB para ver old_role, new_role, IP address, user agent
  7. Sistema muestra stats de actividad reciente (gráficos de acciones por día)
Flujo alternativo: 2a. Usuario sin permiso manage_users → HTTP 403 Forbidden
3a. Supabase no disponible → Sistema muestra error "Error de conexión a auditoría"
Postcondición: Registro de auditoría completo visible con filtros aplicados

7.4 Procedimientos de Mantenimiento

Backup de Base de Datos

pg_dump -h [SUPABASE_HOST] -U postgres.[REF] -p 6543 -d postgres 
        -t page_visits -t inventario_user_permissions -t inventario_audit_log
        --no-owner --no-acl -f backup_$(date +%Y%m%d).sql

Gestionar Usuarios

  1. Acceder a /admin/users (requiere permiso manage_users)
  2. Agregar usuario: Botón "Agregar Usuario" → Ingresar email y seleccionar rol
  3. Editar usuario: Botón "Editar" → Cambiar rol o permisos
  4. Desactivar usuario: Botón "Eliminar" → Usuario se marca como inactivo (is_active=false)
  5. Ver historial: Botón "Historial de Cambios" → Todos los cambios en audit_log

Actualizar Roles y Permisos (Legacy)

NOTA: Este sistema es legacy y será removido. Usar módulo Admin de usuarios.

  1. Editar archivo roles.json (ya no recomendado)
  2. Commit y push a GitHub
  3. Render auto-redeploy

Rotación de SECRET_KEY

  1. Generar nueva clave: python -c "import secrets; print(secrets.token_hex(32))"
  2. Actualizar variable SECRET_KEY en Render
  3. ⚠️ Todas las sesiones activas se invalidarán

9. Riesgos y Mitigaciones

Riesgo Probabilidad Impacto Mitigación
Odoo XML-RPC no disponible Media Alto • Timeout de 30 segundos
• 3 reintentos automáticos
• Mensaje de error user-friendly
• Monitoreo de logs
Supabase PostgreSQL caído Baja Medio • Fallback automático a SQLite local
• Modo degradado (sin analytics)
• SLA 99.9% de Supabase Pro
Usuarios no autorizados intentan acceder Alta Alto • Whitelist estricta
• OAuth obligatorio
• Logging de intentos fallidos
• Rate limiting (60 req/min)
Session hijacking Baja Alto • Session fingerprinting SHA-256
• Cookies SECURE + HTTPONLY
• Timeout de 15 min
• HTTPS obligatorio
Datos sensibles en logs Media Alto • Logging estructurado sin includes
• Sanitización de logs
• Revisión periódica de logs
• Retención de 30 días
Render downtime Baja Alto • SLA 99.95% de Render Pro
• Health checks automáticos
• Auto-restart en fallos
• Backup de código en GitHub
Cambios en API de Odoo Baja Medio • Versionamiento de API (XML-RPC v2)
• Tests de integración
• Documentación de dependencias
• Contacto con equipo de Odoo
Modificación no autorizada de usuarios/roles Baja Alto • Sistema RBAC estricto (solo admin_full)
• Auditoría completa con IP tracking
• Prevención de auto-eliminación
• Soft delete para recuperación
• Logging de todas las operaciones
Pérdida de trazabilidad en cambios de permisos Baja Medio • AuditLogger con JSONB details
• Retención permanente de auditoría
• Filtros avanzados en audit log
• Timestamps con timezone (America/Lima)

Nota: Los riesgos marcados en azul fueron agregados en la versión 2.1 (Admin Module, abril 2026). En versión 2.2 se eliminó dependencia de archivos planos (whitelist.txt/roles.json).
