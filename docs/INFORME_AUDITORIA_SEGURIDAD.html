Hallazgos Detallados de Seguridad

1. Uso de 'unsafe-inline' en Content Security Policy
Media
Categoría OWASP A03:2021 - Injection (Cross-Site Scripting)
Control ISO 27001 A.14.2.5 - Principios de ingeniería de sistemas seguros
Archivo Afectado app.py (líneas 88-90)

📋 Descripción

La Content Security Policy (CSP) configurada permite scripts y estilos inline mediante las directivas 'unsafe-inline'. Esto reduce significativamente la efectividad de CSP como mecanismo de defensa contra ataques XSS, ya que permite la ejecución de código JavaScript arbitrario inyectado en la página.

ISO/IEC 27001:2022 - Control A.14.2.5: Los principios de ingeniería de sistemas seguros requieren implementar controles que minimicen la superficie de ataque. El uso de 'unsafe-inline' amplía innecesariamente esta superficie.

💻 Código Vulnerable

csp = { 'script-src': [ "'self'", "'unsafe-inline'", # ⚠️ VULNERABLE: Permite scripts inline 'https://cdn.jsdelivr.net', 'https://cdnjs.cloudflare.com' ], 'style-src': [ "'self'", "'unsafe-inline'", # ⚠️ VULNERABLE: Permite estilos inline 'https://cdn.jsdelivr.net', 'https://fonts.googleapis.com' ] }

✅ Remediación Recomendada

Solución 1 (Recomendada): Implementar nonces o hashes para scripts y estilos específicos.

# En app.py - Usar nonces generados por Talisman csp = { 'script-src': [ "'self'", # Remover 'unsafe-inline' - Talisman genera nonces automáticamente 'https://cdn.jsdelivr.net', 'https://cdnjs.cloudflare.com' ], 'style-src': [ "'self'", # Usar nonces para estilos inline críticos 'https://cdn.jsdelivr.net', 'https://fonts.googleapis.com' ] } # Talisman ya tiene content_security_policy_nonce_in=['script-src'] configurado # En templates/base.html - Usar el nonce generado <script nonce="{{ csp_nonce() }}"> // Tu código JavaScript inline aquí </script>

Solución 2 (Alternativa): Mover todos los scripts y estilos inline a archivos externos.

// Crear static/js/dashboard.js function initDashboard() { // Código que antes estaba inline en dashboard.html const chart = new Chart(ctx, config); } // En templates/dashboard.html <script src="{{ url_for('static', filename='js/dashboard.js') }}"></script>

Impacto de la Remediación: Mejora la puntuación de seguridad en +8 puntos y elimina un vector de ataque XSS significativo.

2. Gestión de Secrets sin Rotación Automática
Media
Categoría OWASP A02:2021 - Cryptographic Failures
Control ISO 27001 A.8.24 - Gestión de claves criptográficas
Archivos Afectados .env, app.py, odoo_manager.py

📋 Descripción

Las claves secretas (SECRET_KEY, GOOGLE_CLIENT_SECRET, ODOO_PASSWORD) se almacenan en un archivo .env sin mecanismo de rotación automática. Si un atacante obtiene acceso a este archivo o si un secret se compromete, no existe un proceso automatizado para regenerar y actualizar las credenciales. Además, config.py define SECRET_ROTATION_DAYS = 90 pero no se implementa la lógica de rotación.

ISO/IEC 27001:2022 - Control A.8.24: La información de autenticación secreta debería ser gestionada mediante un proceso formal que incluya cambio regular de claves, protección durante almacenamiento y transmisión, y procedimientos de emergencia para casos de compromiso.

💻 Situación Actual

# .env (ejemplo) SECRET_KEY=super_secret_key_12345 GOOGLE_CLIENT_SECRET=GOCSPX-xxx... ODOO_PASSWORD=password123 # config.py - Configuración sin implementación class SecurityConfig: SECRET_ROTATION_DAYS: Final[int] = 90 # ⚠️ Definido pero no usado

✅ Remediación Recomendada

Solución 1 (Recomendada): Implementar rotación automática con AWS Secrets Manager, Azure Key Vault o HashiCorp Vault.

# secrets_manager.py - Nuevo módulo para gestión de secrets import boto3 from datetime import datetime, timedelta from logging_config import get_logger logger = get_logger('secrets_manager') class SecretsManager: def __init__(self): self.client = boto3.client('secretsmanager', region_name='us-east-1') self.rotation_days = Config.Security.SECRET_ROTATION_DAYS def get_secret(self, secret_name: str) -> str: """Obtiene secret desde AWS Secrets Manager con cache""" try: response = self.client.get_secret_value(SecretId=secret_name) return response['SecretString'] except Exception as e: logger.error(f"Error obteniendo secret {secret_name}: {e}") raise def check_rotation_needed(self, secret_name: str) -> bool: """Verifica si un secret necesita rotación""" metadata = self.client.describe_secret(SecretId=secret_name) last_rotated = metadata.get('LastRotatedDate') if not last_rotated: return True days_since_rotation = (datetime.now() - last_rotated).days return days_since_rotation >= self.rotation_days # En app.py - Usar SecretsManager from secrets_manager import SecretsManager secrets = SecretsManager() app.secret_key = secrets.get_secret('flask_secret_key')

Solución 2 (Implementación Rápida): Script de verificación y alerta de rotación.

# scripts/check_secret_age.py import os from datetime import datetime, timedelta from pathlib import Path def check_env_file_age(): env_path = Path('.env') if not env_path.exists(): return modified_time = datetime.fromtimestamp(env_path.stat().st_mtime) days_old = (datetime.now() - modified_time).days if days_old >= 90: print(f"⚠️ ALERTA: .env tiene {days_old} días sin modificar") print("🔄 Considera rotar SECRET_KEY y otras credenciales") # Enviar notificación por email/Slack if __name__ == '__main__': check_env_file_age() # Ejecutar con cron job cada semana: 0 9 * * MON python scripts/check_secret_age.py

3. Whitelist de Usuarios en Texto Plano
Media
Categoría OWASP A01:2021 - Broken Access Control
Control ISO 27001 A.9.2.1 - Registro y baja de usuarios
Archivo Afectado odoo_manager.py (líneas 153-183)

📋 Descripción

La lista blanca de usuarios autorizados se almacena en whitelist.txt (desarrollo) o en la variable de entorno WHITELIST_EMAILS (producción) sin ningún tipo de encriptación o hashing. Si un atacante obtiene acceso de lectura a estos archivos o variables, puede conocer todos los usuarios autorizados del sistema, facilitando ataques de ingeniería social o phishing dirigido.

ISO/IEC 27001:2022 - Control A.9.2.1: El registro de usuarios debe ser gestionado mediante un proceso formal que proteja la información de identidad de los usuarios. La lista de usuarios autorizados debe considerarse información sensible.

💻 Código Actual

# whitelist.txt (ejemplo) jonathan.cerda@agrovetmarket.com maria.lopez@agrovetmarket.com juan.perez@agrovetmarket.com # ⚠️ Emails visibles en texto plano # odoo_manager.py def _load_whitelist(self): whitelist = set() env_whitelist = os.getenv('WHITELIST_EMAILS') if env_whitelist: for email in env_whitelist.split(','): whitelist.add(email.lower()) # ⚠️ Sin protección return whitelist

✅ Remediación Recomendada

Solución 1 (Recomendada): Almacenar hashes de emails en lugar de emails en texto plano.

# scripts/generate_whitelist_hashes.py - Script para generar hashes import hashlib def hash_email(email: str) -> str: """Genera hash SHA-256 de un email""" return hashlib.sha256(email.lower().encode()).hexdigest() emails = [ 'jonathan.cerda@agrovetmarket.com', 'maria.lopez@agrovetmarket.com' ] # Generar whitelist_hashes.txt with open('whitelist_hashes.txt', 'w') as f: for email in emails: f.write(f"{hash_email(email)}\n") # odoo_manager.py - Modificar para usar hashes def _load_whitelist(self): """Carga hashes de emails autorizados""" whitelist_hashes = set() whitelist_path = os.path.join(os.path.dirname(__file__), 'whitelist_hashes.txt') if os.path.exists(whitelist_path): with open(whitelist_path, 'r') as f: for line in f: hash_value = line.strip() if hash_value and not hash_value.startswith('#'): whitelist_hashes.add(hash_value) return whitelist_hashes def is_user_authorized(self, username: str) -> bool: """Verifica si el hash del email está en la whitelist""" if not self.whitelist: return True email_hash = hashlib.sha256(username.lower().encode()).hexdigest() return email_hash in self.whitelist

Solución 2 (Alternativa): Mover whitelist a base de datos con encriptación.

# Crear tabla en PostgreSQL CREATE TABLE authorized_users ( id SERIAL PRIMARY KEY, email_hash VARCHAR(64) UNIQUE NOT NULL, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, created_by VARCHAR(255) ); CREATE INDEX idx_email_hash ON authorized_users(email_hash); # odoo_manager.py - Consultar desde DB def is_user_authorized(self, username: str) -> bool: import psycopg2 email_hash = hashlib.sha256(username.lower().encode()).hexdigest() try: with psycopg2.connect(os.getenv('DATABASE_URL')) as conn: with conn.cursor() as cur: cur.execute("SELECT 1 FROM authorized_users WHERE email_hash = %s", (email_hash,)) return cur.fetchone() is not None except Exception as e: logger.error(f"Error verificando autorización: {e}") return False

Beneficios Adicionales:

  • ✅ Protección contra exposición de usuarios autorizados
  • ✅ Auditoría de cambios en whitelist (quién agregó a quién)
  • ✅ Facilita gestión centralizada de accesos
  • ✅ Cumple mejor con GDPR/protección de datos personales

Control ISO 27001 A.14.2.8 - Pruebas de seguridad del sistema

El rate limiting actual usa Flask-Limiter con key_func=get_remote_address, lo que significa que los límites se aplican solo por dirección IP. 5. Falta de Tokens CSRF Explícitos
Baja
Categoría OWASP A01:2021 - Broken Access Control (CSRF)
Control ISO 27001 A.14.2.1 - Política de desarrollo seguro
Archivos Afectados app.py, templates/*.html

📋 Descripción

Aunque la aplicación configura SESSION_COOKIE_SAMESITE = 'Lax', lo cual proporciona cierta protección contra CSRF, no implementa tokens CSRF explícitos. Esto constituye una defensa en profundidad incompleta. Además, SameSite=Lax no protege contra ataques CSRF en navegadores antiguos o en ciertos escenarios de navegación top-level.

ISO/IEC 27001:2022 - Control A.14.2.1: La política de desarrollo seguro debe incluir controles de seguridad establecidos como tokens anti-CSRF para operaciones sensibles que cambian estado.

💻 Configuración Actual

# config.py - Solo SameSite configurado class SessionConfig: COOKIE_SAMESITE: Final[str] = 'Lax' # ⚠️ Protección parcial # templates/inventory.html - Sin token CSRF <form method="POST" action="/inventory"> <!-- ⚠️ No hay campo csrf_token --> <input type="text" name="search_term"> <button type="submit">Buscar</button> </form>

✅ Remediación Recomendada

Implementar Flask-WTF con tokens CSRF

# requirements.txt - Agregar Flask-WTF==1.2.1 # app.py - Configurar CSRF protection from flask_wtf.csrf import CSRFProtect app = Flask(__name__) app.secret_key = os.getenv('SECRET_KEY') # Habilitar protección CSRF global csrf = CSRFProtect(app) # Opcionalmente, excluir ciertos endpoints (como /api/keep-alive) @app.route('/api/keep-alive', methods=['POST']) @csrf.exempt # Excluir solo endpoints específicos def keep_alive(): ... # templates/base.html - Agregar meta tag para AJAX <head> <!-- Para requests AJAX --> <meta name="csrf-token" content="{{ csrf_token() }}"> </head> # templates/inventory.html - Agregar token en formularios <form method="POST" action="/inventory"> <input type="hidden" name="csrf_token" value="{{ csrf_token() }}"/> <input type="text" name="search_term"> <button type="submit">Buscar</button> </form> # static/js/script.js - Para requests AJAX const csrfToken = document.querySelector('meta[name="csrf-token"]').getAttribute('content'); fetch('/api/endpoint', { method: 'POST', headers: { 'Content-Type': 'application/json', 'X-CSRFToken': csrfToken // ✅ Incluir token }, body: JSON.stringify(data) });

Alternativa: Double-Submit Cookie Pattern

# app.py - Implementación manual de double-submit import secrets @app.before_request def generate_csrf_token(): if 'csrf_token' not in session: session['csrf_token'] = secrets.token_hex(32) @app.before_request def validate_csrf_token(): if request.method in ['POST', 'PUT', 'DELETE', 'PATCH']: token = session.get('csrf_token') submitted_token = request.form.get('csrf_token') or \ request.headers.get('X-CSRF-Token') if not token or token != submitted_token: abort(403, description="CSRF token inválido")

✅ Remediación Recomendada

Implementar sanitización de PII en logs

# logging_config.py - Agregar función de sanitización import hashlib def sanitize_email(email: str) -> str: """ Sanitiza email para logs manteniendo trazabilidad. Ejemplo: jonathan.cerda@agrovetmarket.com → j***a@a***t.com [hash:a1b2c3] """ if not email or '@' not in email: return '[invalid]' local, domain = email.split('@') domain_parts = domain.split('.') # Ofuscar local part if len(local) > 2: local_sanitized = local[0] + '***' + local[-1] else: local_sanitized = '***' # Ofuscar domain if len(domain_parts) > 1: domain_sanitized = domain_parts[0][0] + '***' + domain_parts[0][-1] + '.' + domain_parts[-1] else: domain_sanitized = '***' # Agregar hash corto para trazabilidad email_hash = hashlib.sha256(email.encode()).hexdigest()[:6] return f"{local_sanitized}@{domain_sanitized} [hash:{email_hash}]" def sanitize_ip(ip: str) -> str: """ Sanitiza IP manteniendo subnet. Ejemplo: 192.168.1.100 → 192.168.***.*** [hash:x7y8z9] """ if not ip: return '[unknown]' parts = ip.split('.') if len(parts) == 4: subnet = f"{parts[0]}.{parts[1]}" ip_hash = hashlib.sha256(ip.encode()).hexdigest()[:6] return f"{subnet}.***. *** [hash:{ip_hash}]" return f"[hash:{hashlib.sha256(ip.encode()).hexdigest()[:6]}]" # app.py - Usar funciones de sanitización from logging_config import sanitize_email, sanitize_ip logger.warning( "[SECURITY] Posible session hijacking | user=%s | ip=%s | path=%s", sanitize_email(session.get('username', '')), # ✅ Email sanitizado sanitize_ip(request.remote_addr), # ✅ IP sanitizada request.path ) logger.info( "[SECURITY] Sesión expirada | user=%s | inactivo=%.1f min", sanitize_email(session.get('username', '')), # ✅ Email sanitizado inactive_time.total_seconds() / 60 )

Media
Categoría OWASP A05:2021 - Security Misconfiguration

Los archivos estáticos (CSS, JS, imágenes) servidos por Flask no incluyen headers de cache control ni de seguridad específicos, lo que podría permitir ataques de cache poisoning o servir contenido desactualizado.

✅ Remediación Recomendada

# app.py - Configurar cache y seguridad para estáticos @app.after_request def add_static_headers(response): if request.endpoint == 'static': # Cache público para archivos estáticos (1 año) response.headers['Cache-Control'] = 'public, max-age=31536000, immutable' # Evitar MIME-sniffing response.headers['X-Content-Type-Options'] = 'nosniff' else: # No cache para páginas dinámicas response.headers['Cache-Control'] = 'no-store, no-cache, must-revalidate, private' response.headers['Pragma'] = 'no-cache' response.headers['Expires'] = '0' return response
